# P20-01 — Amazon SES Configuration

## Purpose

Set up Amazon SES as the production email sending service for Kontax. SES requires domain verification, DKIM/SPF DNS records, and IAM credentials before any email can be sent. This ticket covers the setup steps, documents them as runbook entries, and validates the configuration at application startup.

## Background

Amazon SES operates in a sandbox by default — only verified email addresses can receive email. Production lift requires a sending limit increase request to AWS (typically approved within 24 hours if the request is reasonable). Starting in sandbox allows safe testing without reaching real user inboxes.

## Scope

**In scope:**
- Domain verification in SES (CNAME records for DKIM; TXT record for domain verification)
- SPF record addition to the sending domain's DNS
- IAM user/role with `ses:SendEmail` and `ses:SendRawEmail` permissions
- `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SES_REGION`, `EMAIL_FROM` env vars added to `.env.example` and validated in `src/env.js`
- Application startup check: if SES credentials are absent, log a warning and use console fallback (not an error — dev should work without SES)
- SNS topic creation for bounce/complaint notifications (used in P20-10)
- Runbook document: step-by-step SES setup guide

**Out of scope:**
- The `sendEmail` abstraction (P20-02)
- Sandbox → production lift request (this is an AWS console action, not code — document it)

## Design / Implementation Spec

### DNS records required

The following DNS records must be added to the Kontax sending domain (e.g. `kontax.app`):

**DKIM (three CNAME records):**
```
{token1}._domainkey.kontax.app  CNAME  {token1}.dkim.amazonses.com
{token2}._domainkey.kontax.app  CNAME  {token2}.dkim.amazonses.com
{token3}._domainkey.kontax.app  CNAME  {token3}.dkim.amazonses.com
```
Tokens are generated by SES and shown in the Verified Identities console. DKIM propagation takes up to 72 hours.

**SPF (add to existing TXT record or create new):**
```
TXT  kontax.app  "v=spf1 include:amazonses.com ~all"
```

**DMARC (recommended):**
```
TXT  _dmarc.kontax.app  "v=DMARC1; p=quarantine; rua=mailto:dmarc@kontax.app"
```

### IAM policy

Create an IAM user `kontax-ses-sender` with the following inline policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["ses:SendEmail", "ses:SendRawEmail"],
      "Resource": "arn:aws:ses:{region}:{account-id}:identity/kontax.app"
    },
    {
      "Effect": "Allow",
      "Action": ["sns:Publish"],
      "Resource": "arn:aws:sns:{region}:{account-id}:kontax-email-events"
    }
  ]
}
```

Generate access keys for this user and store as `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY`.

### SNS topic for bounce/complaint handling

Create an SNS topic `kontax-email-events` in the same region. Configure SES to publish bounce and complaint notifications to this topic. Add a subscription to the topic pointing at `{APP_URL}/api/ses/events` (P20-10 implements this endpoint).

### Application startup validation

In `src/env.js`, mark the SES vars as optional with a server-side startup check:

```typescript
// src/server/email.ts — called at module load
export const SES_CONFIGURED =
  !!process.env.AWS_ACCESS_KEY_ID &&
  !!process.env.AWS_SECRET_ACCESS_KEY &&
  !!process.env.AWS_SES_REGION &&
  !!process.env.EMAIL_FROM;

if (!SES_CONFIGURED) {
  console.warn("[email] SES not configured — emails will be logged to console only");
}
```

This is checked by `sendEmail` (P20-02) before attempting any SES API call.

### Sandbox → production lift

To send email to unverified addresses (i.e., real users), request a sending limit increase in the AWS console:
- Go to **SES → Account dashboard → Request production access**
- Use case description: transactional emails for a SaaS contacts product (verification, password reset, share notifications)
- Requested sending rate: start with 14 emails/second (the minimum)
- AWS reviews within 24 hours in most cases

Document this as a runbook step. No code changes are needed — the SES SDK works identically in sandbox and production.

## Acceptance Criteria

- Domain is verified in SES; DKIM CNAME records are added to DNS.
- SPF and DMARC records are added.
- IAM user `kontax-ses-sender` exists with the correct policy.
- `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SES_REGION`, `EMAIL_FROM` are documented in `.env.example`.
- `SES_CONFIGURED` is `false` when env vars are absent; app starts normally with a console warning.
- SNS topic `kontax-email-events` is created and SES is configured to publish bounces/complaints to it.
- A test email can be sent from the AWS console to a verified address in sandbox mode.

## Risks and Open Questions

- **DNS propagation:** DKIM CNAME records can take up to 72 hours to propagate. Start this ticket early — it can be done while other Phase 20 tickets are in progress.
- **Sending domain vs FROM address:** SES verifies the domain or the specific FROM address. Verifying the domain (`kontax.app`) is more flexible than verifying individual addresses. Use domain verification.
- **Credential rotation:** IAM access keys should be rotated periodically. The SES-specific IAM user limits the blast radius of a key leak to email sending only. Document the rotation procedure in the runbook.
