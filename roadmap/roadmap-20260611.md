# Kontax Roadmap — 2026-06-11

## Overview

This roadmap covers the next batch of product and platform work following the completion of Phases 1–17. It is organized into thirteen phases addressing: core account management and infrastructure, Stripe billing, email delivery, admin tooling, notification/security alerting, sync advanced settings, mobile optimization and PWA, import/export UX polish, public surfaces, onboarding and SEO, Google/Outlook OAuth sync, power user productivity, compliance/developer access, and the public contact card.

Each phase has a design brief ticket where a meaningful UI surface is involved.

---

| DB-05 | Design brief — notification settings and suspicious activity surfaces | P1 | P22-02, P22-05 |

### Notes
- P22-04 detection rules run as a post-mutation hook in the server actions that emit ActivityEvents (P10-02). Keep detection synchronous for immediate alerting; move to a background job if latency becomes a concern.
- "Wasn't me" flow (P22-06) is security-critical: it must invalidate all sessions including the current one and force a password reset before the account is usable again.
- Notification categories to consider: security alerts, contact sharing, sync status, billing, product updates, birthday/anniversary reminders. Security alerts should default to all channels on; product updates should default off.
- P22-09 runs as a daily background job (same cron infrastructure as P18-09's deletion job). It reads each user's contacts for `birthday` and `significantDates` values, computes upcoming matches within the lead-time window, and enqueues a `NotificationEvent` if one hasn't already been sent this year for that date.
- P22-11 iCal feed: the `.ics` URL includes a per-user `calToken` query param (random, revocable, stored on User — not the session token) so the URL can be shared with calendar apps without exposing account credentials. Revoke via settings. Each VEVENT has `RRULE:FREQ=YEARLY` so birthdays recur automatically.

- P24-01 must be run against real devices, not just browser DevTools — iOS Safari and Android Chrome have known divergences in input handling, scroll behaviour, and viewport height.

- The PWA replaces the need for a native app for now. App Store / Play Store distribution is a future decision; a PWA can be submitted to both stores via tooling like PWABuilder when the time comes.

- P28-01 smart lists use the same filter state object that Phase 16's URL params encode. Persist the serialised filter state as JSON in `SavedFilter.filterState`. Rehydrate on navigation.

- P28-04 bulk edit extends the existing bulk-select infrastructure from P16-04. The "move to book" action is only possible after P28-03 ships.
- P28-05 keyboard shortcuts: use a lightweight library (e.g. `hotkeys-js`) or implement with a `useEffect` event listener. Document all shortcuts in the overlay modal and in a `/keyboard-shortcuts` help page.
- P28-07 full-text search: add a `searchVector tsvector` generated column to the `Contact` table, populated from `fullName`, `email`, `phone`, `company`, `notes`, `jobTitle`, and the JSON fields coerced to text. Use Postgres `GIN` index for performance. The search API replaces the current `ILIKE` approach and returns results ranked by `ts_rank`. This is a P0 because basic name/email search already exists — full-text is the gap that makes power users reach for a spreadsheet instead.

### Notes
- P29-01 data export must include everything Kontax holds. The ZIP structure: `contacts.vcf`, `contacts.csv`, `activity.json`, `billing-summary.txt`, `account.json`. This is the same export offered as a pre-deletion suggestion in P18-09.
- P29-05 API tokens reuse the `AppPassword` model pattern from P9-02 (32-byte random token, SHA-256 hash stored, shown once, revocable). Add `scope` and `lastUsedAt` fields. Do not reuse `AppPassword` directly — create a separate `ApiToken` model to keep concerns distinct.

- `SourceType.API` already exists in the schema — contacts created or updated via the API will be correctly attributed.
- Rate limits: 1,000 requests/hour for read-only tokens; 200 requests/hour for read-write tokens. These are starting values and can be adjusted per plan tier.

### Design Brief

- P30-01 username: reserve common words (`admin`, `api`, `kontax`, `help`, `pricing`, `login`, `register`, `settings`, `u`, `c`, `dav`, etc.) at the DB level. Enforce at the application layer before insert.
- P30-02 is server-rendered (Next.js App Router server component) for SEO — the public card must be crawlable. Add `<meta name="robots" content="index, follow">` and structured data (`Person` JSON-LD schema) to the card page.

- P30-06 analytics: simple view counter stored on `User.publicCardViews Int @default(0)` and a `PublicCardView` log table (userId, viewedAt, referrer). No third-party analytics — keep it in Kontax's own database.

## Suggested Build Order

Phase dependencies:
- **18** (account/auth) must land before 19 (billing needs verified email), 20 (SES needs email verification), 22 (suspicious activity needs session model), and 26 (login/register pages need all Phase 18 auth features wired in).
- **19** (Stripe) can run in parallel with 20 once 18 is done.
- **20** (SES) depends on 18 (email verification logic); templates for notifications depend on 22 being designed.
- **21** (admin) depends on 18 (role model) and ideally 19 (plan override needs Stripe).
- **22** (notifications) depends on 18 (sessions). Birthday reminders (P22-09) can run independently of the suspicious-activity work.
- **23** (sync settings) is independent — can start anytime.
- **24** (mobile/PWA) is mostly independent — can run in parallel. P24-08 (PWA) depends on P24-07 (performance). The PWA is the mobile delivery mechanism; no native app.
- **25** (import mapping) is independent — can run in parallel with any phase.
- **26** (public surfaces/onboarding/SEO) depends on 18 for login/register pages (all auth features wired). Landing page P26-01 and SEO tickets P26-07–P26-11 are independent of 18 and can start earlier.
- **27** (Google/Outlook sync) depends on 18 (OAuth provider infrastructure from P18-08) and Phase 5/7 (existing sync model). Deduplication (P27-08) depends on Phase 4 merge engine (already done).
- **28** (power user productivity) depends on P18-11 (address book model) for P28-03, and Phase 16 (contacts list) for smart lists and bulk edit. Otherwise independent.
- **29** (compliance/developer) depends on 18 (session model for API tokens), 19 (billing history for data export), and 20 (email for export-ready notification). Can start on P29-05/06 (API tokens) independently.

Recommended sequence:
- **Foundation:** 18 (with 26 landing page + SEO in parallel)
- **Core product:** 19 + 20 + 26 (login/register/onboarding) in parallel
- **Platform:** 21 + 22 + 23 in parallel
- **Growth:** 24 + 25 + 27 in parallel
- **Depth:** 28 + 29 in parallel
- **Growth:** 30 (public contact card) — depends on 26 (onboarding, where username is claimed) and 29 (API, which P30-04 can reuse). Can start P30-02/03 (page + visibility) independently once P30-01 (username) is done.

> **Excluded from this roadmap:** Native iOS/Android apps — Kontax ships as a PWA (Phase 24) and relies on CardDAV for native device sync (Phase 9). Dark mode (deferred).
